1 Domaine d'application

Ce manuel d'utilisation spécifie la connexion physique, la liaison de communication et les spécifications techniques d'application entre l'interrupteur automatique de transfert et la station maître. Ce manuel s'applique aux produits d'interrupteurs automatiques de transfert de type T (prenant en charge la communication) des 8 séries suivantes : NXZMN, NXZHMN, NZ5M, NZ5HM, NXZBN, NXZHBN, NZ5B, NZ5HB.

2 Aperçu du protocole

Le protocole Modbus est un protocole de bus industriel basé sur le modèle ISO/OSI (7 couches), mais il n'utilise que 3 des 7 couches (couche physique, couche liaison de données et couche application), simplifiant ainsi le modèle de protocole et réduisant la difficulté d'utilisation.

2.1 Définitions et terminologie

Les termes suivants s'appliquent à ce manuel d'utilisation.

• Modèle d'interconnexion de systèmes ouverts (OSI)
Norme établie par l'Organisation internationale de normalisation (ISO) en 1984, visant à fournir une base commune et un cadre standard pour l'interconnexion d'ordinateurs de différents fabricants.

• Couche physique
Première couche du modèle d'interconnexion de systèmes ouverts (OSI), elle fournit la liaison physique pour une transmission transparente des communications.

• Couche liaison de données
Deuxième couche du modèle d'interconnexion de systèmes ouverts (OSI), elle fournit des services de transmission d'informations transparents et fiables entre nœuds adjacents.

• Couche application
Septième couche du modèle d'interconnexion de systèmes ouverts (OSI), elle implémente les fonctions spécifiques d'opération et d'échange d'informations.

• Trame
Dans les données et les communications numériques, structure d'information spécifique composée de plusieurs bits ou champs prédéterminés selon une certaine norme. Les données sont transmises sur le réseau en petites unités appelées trames (Frame), composées de plusieurs parties exécutant différentes fonctions.

2.3 Couche liaison de données

Mode de transmission : mode semi-duplex maître-esclave (interrogation maître, réponse esclave).
Type de protocole : Le protocole de communication utilise le mode Modbus-RTU.

2.3.1 Format de transmission série

1 trame de données contient : 1 bit de démarrage, 8 bits de données, 1 bit de parité, 1 bit d'arrêt.

Démarrage Données Parité Arrêt
1 12345678 1 1

2.3.2 Format du paquet de données (plusieurs trames)

Début Adresse Fonction Données Contrôle Fin
T3.5 8 bits 8 bits n×8 bits 16 bits T3.5

En mode RTU, l'information nécessite au moins un temps de silence de 3,5 caractères (ou trames) au début. Ce temps de silence (T3.5 dans le tableau ci-dessus) se calcule facilement en fonction du débit en bauds utilisé. Ce délai doit être pris en compte lors de l'utilisation du microcontrôleur UART pour créer un protocole de communication. Si un logiciel de configuration ou un système DCS est utilisé, et que la couche logicielle sous-jacente est déjà bien faite, il n'est pas nécessaire de le prendre en compte lors de l'utilisation.

L'intervalle de temps entre plusieurs paquets de données doit être d'au moins 200 millisecondes. Une période d'envoi inférieure à 100 millisecondes peut entraîner une augmentation du taux de perte de paquets.

2.4 Couche application

La couche application analyse le paquet de données (comprenant le code d'adresse, le code de fonction, le champ de données, le code de vérification, etc.) pour atteindre l'objectif d'échange de données. Format comme suit :

Code d'adresse Code de fonction Adresse de début du registre Nombre de variables Code de vérification CRC
  Octet haut Octet bas Octet haut
03H 03H 00H 06H 00H

2.4.1 Code d'adresse

Le code d'adresse se trouve au début du paquet de données, composé de 8 bits (de 1 à 247). Le code d'adresse par défaut est 3.

Le code d'adresse indique l'adresse de l'équipement esclave spécifié par l'utilisateur ; cet équipement esclave recevra les données du maître qui y est connecté.

L'adresse de chaque équipement esclave dans un même réseau doit être unique. Seul l'esclave adressé répondra à la requête contenant cette adresse. Lorsque l'esclave renvoie une réponse, l'adresse esclave dans la réponse indique au maître quel esclave communique avec lui. Si le maître envoie une adresse de communication erronée à l'esclave, l'esclave ne répondra pas.

2.4.2 Code de fonction

Le code de fonction indique à l'esclave adressé la fonction à exécuter. Les codes de fonction envoyés par le maître sont présentés dans le tableau suivant :

Code de fonction Définition Action
03H Lecture du registre de données Lit la valeur de données d'un ou plusieurs registres
06H Écriture d'un seul registre Écrit des données dans un registre

Voir la section 3 Codes de fonction pour plus de détails.

2.4.3 Champ de données

Le champ de données contient les données nécessaires à l'esclave pour exécuter une fonction spécifique ou les données collectées par l'esclave en réponse à une interrogation. Le contenu de ces données peut être des valeurs numériques, des adresses de référence ou des valeurs limites. Le contenu spécifique se réfère au tableau des registres d'adresses de communication.

Par exemple : si le code de fonction indique à l'esclave de lire un registre, le champ de données doit spécifier à partir de quel registre commencer et combien de données lire. Les adresses et données intégrées varient selon le type et les capacités de l'esclave.

2.4.4 Code de vérification

Ce champ permet au maître et à l'esclave de vérifier les erreurs pendant la transmission. Parfois, en raison du bruit électrique et d'autres interférences, un ensemble de données peut être modifié sur la ligne lors du transfert d'un équipement à un autre. La vérification des erreurs garantit que le maître ou l'esclave ne répond pas aux données modifiées pendant la transmission, améliorant ainsi la sécurité et l'efficacité du système. La vérification des erreurs utilise la méthode de redondance cyclique sur 16 bits.

Le champ de contrôle de redondance cyclique (CRC) occupe deux octets et contient une valeur binaire 16 bits. Lors de la transmission, l'octet de poids faible est envoyé en premier, suivi de l'octet de poids fort.

La valeur CRC est calculée par l'équipement émetteur, puis ajoutée à la trame de données. L'équipement récepteur recalcule la valeur CRC lors de la réception des données, puis la compare à la valeur du champ CRC reçu. Si les deux valeurs ne sont pas égales, une erreur s'est produite.

La méthode de génération CRC16 est décrite à l'Annexe A - Principe de génération CRC-16.

Code de fonction Définition Action

Lorsque le maître envoie une instruction avec le code de fonction 03H, dans des conditions normales, le code de fonction de réponse de l'esclave est 03H ; en cas d'erreur, l'esclave répond avec 83H.

Lorsque le maître envoie une instruction avec le code de fonction 06H, dans des conditions normales, le code de fonction de réponse de l'esclave est 06H ; en cas d'erreur, l'esclave répond avec 86H.

Lorsque le maître envoie une trame d'informations erronée à l'esclave (adresse de communication erronée, code de vérification erroné, ou trame incomplète, etc.), l'esclave ne répond pas.

3.3 Gestion des erreurs

Lorsque le maître envoie une trame d'informations erronée à l'esclave (code de fonction erroné, adresse de registre erronée, données erronées ou non modifiables, etc.), l'esclave renvoie les informations d'erreur correspondantes.
Si le maître envoie une instruction erronée avec le code de fonction 03H, le code de fonction de réponse de l'esclave est 83H.
Si le maître envoie une instruction erronée avec le code de fonction 06H, le code de fonction de réponse de l'esclave est 86H.

Causes d'erreur spécifiques :
01 : Données illégales
02 : Erreur d'adresse de registre
03 : Erreur de longueur de données

4.6 Registre de transfert forcé

La commande de transfert forcé ne peut être utilisée qu'en état de contrôle à distance. C'est-à-dire qu'il faut d'abord envoyer une commande de contrôle (adresse : 0x2800) pour entrer en état de contrôle à distance.

Commande de transfert forcé (adresse : 0x2700) Écriture seule  
0x0000 Transfert forcé vers position alimentation I Valable uniquement en "état de contrôle à distance"
0x00aa Transfert forcé vers position alimentation II 
0x00ff Transfert forcé vers position double ouverture 

Lors d'un transfert forcé, c'est-à-dire lors de la fermeture sur une source d'alimentation particulière, le produit ne peut exécuter l'action que si la tension de cette source est normale.

Pour les séries NXZ(H)MN\NZ5(H)M, en contrôle à distance, le verrouillage à distance doit être désactivé (désactivé par défaut). C'est-à-dire appuyer sur la touche de réglage sur le panneau d'affichage du contrôleur, appuyer sur la touche "√" pour tourner les pages, entrer dans le sous-menu "A-F", et régler sur OFF.

Exemples :

1. Entrer en contrôle à distance : Envoi maître [03 06 28 00 00 04 80 4B] Réponse esclave [03 06 28 00 00 04 80 4B]
2. Transfert forcé vers double ouverture : Envoi maître [03 06 27 00 00 FF C2 DC] Réponse esclave [03 06 27 00 00 FF C2 DC]
3. Transfert forcé vers alimentation I : Envoi maître [03 06 27 00 00 00 08 29 C] Réponse esclave [03 06 27 00 00 00 08 29 C]
4. Transfert forcé vers alimentation II : Envoi maître [03 06 27 00 00 AA 02 E3] Réponse esclave [03 06 27 00 00 AA 02 E3]
5. Quitter le contrôle à distance : Envoi maître [03 06 28 00 00 00 81 88] Réponse esclave [03 06 28 00 00 00 81 88]

5 Méthodes de connexion de communication

5.1 Connexion matérielle

Connecter les fils A, B et GND du bus RS485 respectivement aux bornes (A+), (B-) et (GND) du contrôleur de manière fiable. Ensuite, mettre sous tension normalement l'alimentation I ou l'alimentation II, ou utiliser une alimentation externe pour garantir une alimentation normale du contrôleur.

5.2 Configuration des paramètres de communication

S'assurer que les paramètres de communication de l'ordinateur hôte et du contrôleur sont cohérents. Il y a 6 paramètres de communication au total : 3 paramètres réglables (adresse de communication, débit en bauds, bit de parité) et 3 paramètres fixes (bits de données, bits d'arrêt, contrôle de flux). Pour les séries NZX(H)BN, NZ5(H)B, en raison de l'absence de panneau d'affichage, il n'est pas possible d'utiliser le menu pour configurer les paramètres de communication. Il n'est donc généralement pas recommandé de modifier les paramètres de communication. Si une modification est nécessaire, la connexion de communication doit d'abord être établie, puis les paramètres peuvent être modifiés à l'aide d'une instruction d'écriture Modbus ; pour restaurer les paramètres de communication par défaut, un signal d'incendie peut être appliqué pendant plus de 60 secondes.

5.3 Dépannage des anomalies de communication

(1) Vérifier le matériel : Vérifier si les fils A et B du bus RS485 sont desserrés ou inversés sur les bornes (A+), (B-) du contrôleur ; vérifier si l'alimentation est normale.
(2) Vérifier si les paramètres de communication du contrôleur de l'interrupteur automatique de transfert sont correctement configurés (doivent être cohérents avec ceux de l'ordinateur hôte). Pour les séries NZX(H)MN, NZ5(H)M, les paramètres de communication réglables peuvent être configurés sur le panneau d'affichage du produit. Pour les séries NZX(H)BN, NZ5(H)B, appliquer un signal d'incendie pendant plus de 60 secondes pour restaurer les paramètres de communication par défaut.
(3) Vérifier si les paramètres du port série dans l'hôte de communication (débogueur) sont correctement configurés (doivent être cohérents avec les paramètres du contrôleur).
(4) Vérifier si l'adaptateur de communication (convertisseur RS485) est endommagé (tester avec un nouveau).
(5) Si tous les points ci-dessus sont en ordre, contacter notre société pour une analyse plus approfondie.

Annexe A - Principe de génération CRC-16

Lors de l'opération CRC, on préinitialise d'abord un registre 16 bits à tout 1 (valeur FFFFh), puis on opère continuellement avec chaque caractère de 8 bits de la trame de données et la valeur actuelle de ce registre. Seuls les 8 bits de données de chaque octet participent à la génération du CRC ; le bit de démarrage, le(s) bit(s) d'arrêt et l'éventuel bit de parité n'affectent pas le CRC.

Lors de la génération du CRC, chaque caractère de 8 bits est XORé avec le contenu du registre, puis le résultat est décalé vers le bit de poids faible, le bit de poids fort étant complété par "0". Le bit de poids faible (LSB) est décalé et testé. S'il est à 1, le registre est XORé avec une valeur fixe prédéfinie (0A001h) ; s'il est à 0, aucune opération n'est effectuée.

Le traitement ci-dessus est répété jusqu'à ce que 8 opérations de décalage aient été effectuées. Après le décalage du dernier bit (le 8ème), le prochain caractère de 8 bits est XORé avec la valeur actuelle du registre, et les 8 opérations de décalage et de XOR mentionnées ci-dessus sont répétées. Lorsque tous les octets de la trame de données ont été traités, la valeur finale générée est la valeur CRC.

Le processus de génération d'un CRC est le suivant :

a) Préinitialiser un registre 16 bits à la valeur FFFFH (tout à 1), appelé registre CRC.
b) Effectuer un XOR entre le premier caractère de 8 bits de la trame de données et l'octet de poids faible du registre CRC, et stocker le résultat dans le registre CRC.
c) Décaler le registre CRC d'un bit vers la droite, remplir le bit de poids fort avec 0, décaler et tester le bit de poids faible.
d) Si le bit de poids faible est 0 : répéter la troisième étape (décalage suivant).
e) Si le bit de poids faible est 1 : effectuer un XOR entre le registre CRC et une valeur fixe prédéfinie (0A001H).
f) Répéter les étapes c et d jusqu'à 8 décalages. Cela termine le traitement d'un octet complet.
g) Répéter les étapes b à e pour traiter l'octet suivant, jusqu'à ce que tous les octets aient été traités.
h) Le contenu final du registre CRC est la valeur CRC.

Annexe B - Exemple d'application de communication NXZ(H)MN

Réalisation du contrôle à distance pour le transfert vers la position double ouverture

1. Câblage de l'alimentation et de la communication : Tout d'abord, connecter les fils A, B et GND du bus RS485 respectivement aux bornes (A+), (B-) et (G) du contrôleur de l'interrupteur automatique de transfert de manière fiable. Ensuite, mettre sous tension normalement l'alimentation I ou l'alimentation II pour garantir une alimentation normale du contrôleur.
2. Connecter le convertisseur de bus RS485 à l'ordinateur de débogage.
3. S'assurer que les paramètres de communication de l'ordinateur hôte et du contrôleur sont cohérents. Utiliser les paramètres de communication par défaut du contrôleur : débit en bauds 9,6 kbps, parité paire, adresse de communication 3.
4. Ouvrir l'outil de débogage série, configurer le numéro de port série en fonction de la situation réelle : configurer les paramètres du port série comme suit : débit en bauds 9,6 kbps, 1 bit de démarrage, 8 bits de données, parité paire et 1 bit d'arrêt.
5. Test de la fonction de lecture : Envoyer une trame de test (03 03 00 4F 00 01 B4 3F). Si le contrôleur renvoie des données commençant par 03 03 02, la fonction de lecture est normale. Voir la section 4.1 Ensemble de données pour toutes les adresses de registres.
6. Test de la fonction d'écriture : Envoyer une trame de test (03 06 28 00 00 04 80 4B). Si le contrôleur renvoie (03 06 28 00 00 04 80 4B) et que "Manuel" et "Automatique" clignotent simultanément sur l'écran du contrôleur, la communication est normale et le contrôleur entre en "état de contrôle à distance".
7. Test de la fonction d'écriture : Envoyer une trame de test (03 06 27 00 00 FF C2 DC). Le contrôleur renvoie (03 06 27 00 00 FF C2 DC) et le produit se transfère forcément vers la position double ouverture (position intermédiaire). Voir la section 4.6 Registre de transfert forcé.
8. En cas d'anomalie de communication, consulter les points de dépannage à la section 5.3 Dépannage des anomalies de communication.
